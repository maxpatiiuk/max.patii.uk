# Plan: New Reference Plugin (`plugin-v2.ts`)

## Context

The current `plugin.ts` was generated by an LLM that misunderstood the Vite 6
Environments API (uses non-existent `ssrEnvironment.runModule`, mixes execution
contexts, hardcodes `/projects/` prefix). The user wants a clean, well-commented
reference file they can iterate on.

## Architecture Answers

### 1. Environment Creation/Configuration

**Dev:** `client` and `ssr` environments exist by default. No configuration
needed. Use `server.environments.ssr.runner.import()` directly.

**Build:** The plugin's `config` hook adds `environments.ssr` and sets
`builder.buildApp` to sequence: client build → SSR build → prerender.

### 2. resolveId / load / transform

- **`resolveId`** — Not needed for URL routing (that's middleware). Not needed
  for .md files (they have real paths).
- **`load`** — Not needed (.md files exist on disk).
- **`transform`** — The only pipeline hook needed. Converts .md → JS module via
  existing `transformMarkdown()`.

### 3. Module Execution Context

`.md` modules and layout modules **must** be loaded through Vite's SSR runner
(not via direct `import()` in the plugin):

- `.tsx` layout files need Vite's transform pipeline (Lumina JSX compilation)
- Package imports like `@maxpatiiuk/web-components/...` need Vite's resolver
- Vite features (`?raw`, etc.) only work through the runner

**Cross-context TemplateResult composability:** Lit SSR uses duck typing
(`_$litType$` property check), not `instanceof`. So a `TemplateResult` created
by runner-loaded `lit` can be rendered by plugin-loaded `@lit-labs/ssr`. The
plugin composes content + layout templates using `lit/static-html.js` and
`unsafeStatic` for dynamic tag names.

## Type Changes

The current `types.ts` doesn't match the actual collection structure. New shapes
(shown as comments in the file, for user to later move to `types.ts`):

```ts
type PageMetadata = {
  readonly title: string;
  readonly description?: string;
  readonly layout?: string; // override module path
  readonly [key: string]: unknown; // layouts add extra fields
};

type Collection = {
  readonly defaultLayout: string; // module path, e.g. '@maxpatiiuk/web-components/...'
  readonly pages: Record<string, PageMetadata>;
};

type ForgeConfig = {
  readonly siteConfig: SiteConfig;
  readonly collections: Record<string, Collection>;
};
```

`layout` is a **string module path** (not `() => import(...)`) so the plugin
can:

- Load it through the runner during dev
- Add it as an SSR build entry point during build
- Get `.tagName` from the exported Lumina class

## Plugin Hooks

| Hook                | Purpose                                              |
| ------------------- | ---------------------------------------------------- |
| `config()`          | Add `environments.ssr` + `builder.buildApp`          |
| `transform()`       | `.md` → JS module (delegates to `transformMarkdown`) |
| `configureServer()` | Dev middleware + `.md` change → full-reload          |
| `writeBundle()`     | SSR env only: prerender all pages to dist/           |

## Dev Server Flow

```log
HTTP request → middleware
  1. resolveUrlToPage(url, collections) → { collectionName, slug, metadata }
  2. Determine mdPath: root → /src/pages/{slug}.md, other → /src/pages/{collection}/{slug}.md
  3. runner.import(mdPath) → { render }
  4. contentTemplate = render(siteConfig)
  5. layoutPath = metadata.layout ?? collection.defaultLayout
  6. runner.import(layoutPath) → get exported class → .tagName
  7. fullTemplate = staticHtml`<${tag} .layoutData=${metadata}>${content}</${tag}>`
  8. html = renderToString(fullTemplate)
  9. Serve response
```

Uses `isRunnableDevEnvironment()` guard on `server.environments.ssr`.

## Build Flow (Multi-Environment)

```log
config():
  - Collect all .md paths + layout paths → SSR entry points
  - Set environments.ssr.build with those entries
  - Set builder.buildApp: client → ssr → (prerender in writeBundle)

writeBundle() [SSR environment only]:
  - For each collection page:
    1. Import the bundled .md module from SSR output
    2. render(siteConfig) → contentTemplate
    3. Import the bundled layout module → get tagName
    4. Compose with staticHtml + unsafeStatic
    5. renderToString → HTML string
    6. Write to client outDir (e.g., dist/projects/spacetime/index.html)
```

## File: `plugin-v2.ts`

Single file alongside existing `plugin.ts`. Contains:

1. **Type comments** at top showing expected ForgeConfig/Collection/PageMetadata
   shapes
2. **`useStaticSiteForge(config)`** → `Plugin`
3. **`config()` hook** — environments + builder setup
4. **`transform()` hook** — delegates to `transformMarkdown`
5. **`configureServer()` hook** — dev middleware with generic URL resolver
6. **`writeBundle()` hook** — static site generation for SSR build output
7. **Helper: `resolveUrlToPage()`** — URL → collection page (generic, no
   hardcoded paths)
8. **Helper: `renderPage()`** — shared render pipeline used by both dev and
   build
9. **Helper: `getLayoutTagName()`** — loads layout module, extracts `.tagName`
